sudo systemctl restart open5gs-scpd et sudo systemctl restart open5gs-upfd
 ne demarre pas 
 ***************************************
 Si ces commandes :

```bash
sudo systemctl restart open5gs-scpd
sudo systemctl restart open5gs-upfd
```

ne démarrent pas les services, il faut vérifier la cause exacte. Commence par exécuter les commandes suivantes et partage leur sortie :

```bash
sudo systemctl status open5gs-scpd
```

et

```bash
sudo systemctl status open5gs-upfd
```

Ensuite, consulte les journaux :

```bash
sudo journalctl -u open5gs-scpd -n 50 --no-pager
```

```bash
sudo journalctl -u open5gs-upfd -n 50 --no-pager
```

Tu peux également vérifier si les fichiers de configuration sont valides :

```bash
ls -l /etc/open5gs/
```

Vérifie notamment que les fichiers suivants existent :

```text
/etc/open5gs/scp.yaml
/etc/open5gs/upf.yaml
```

Enfin, assure-toi que les services existent bien :

```bash
systemctl list-unit-files | grep open5gs
```

ou

```bash
systemctl | grep open5gs
```

### Causes fréquentes

* Erreur de syntaxe dans `scp.yaml` ou `upf.yaml`.
* Adresse IP configurée qui n'existe pas sur la machine.
* Port déjà utilisé par un autre processus.
* Les paquets Open5GS installés ne contiennent pas `scpd` (certaines versions n'incluent pas le SCP).

Peux-tu me fournir le résultat de :

```bash
sudo systemctl status open5gs-scpd
sudo systemctl status open5gs-upfd
sudo journalctl -u open5gs-scpd -n 20 --no-pager
sudo journalctl -u open5gs-upfd -n 20 --no-pager
```

Dans ce que tu as montré, **il n'y a aucune différence**.

Les deux fichiers contiennent exactement la même configuration :

* `open5gs/build/configs/open5gs/scp.yaml`
* `/etc/open5gs/scp.yaml`

La différence est **leur rôle** :

* `open5gs/build/configs/open5gs/scp.yaml` est un **fichier modèle** généré lors de la compilation d'Open5GS. Il sert de référence et n'est généralement pas utilisé par les services.
* `/etc/open5gs/scp.yaml` est le **fichier réellement utilisé** par `open5gs-scpd`, comme on le voit dans les logs :

```text
Configuration: '/etc/open5gs/scp.yaml'
```

C'est donc **uniquement** `/etc/open5gs/scp.yaml` qu'il faut modifier pour changer le comportement du service.

### En revanche, il y a une incohérence dans ces fichiers

Ils indiquent :

```yaml
scp:
  sbi:
    server:
      - address: 127.0.0.200
        port: 7777
    client:
      nrf:
        - uri: http://127.0.0.10:7777
```

Alors que ton `nrf.yaml` indique :

```yaml
nrf:
  sbi:
    server:
      - address: 127.0.0.200
        port: 7777
```

Le SCP cherche donc le NRF sur **127.0.0.10**, tandis que le NRF écoute sur **127.0.0.200**. Cette incohérence explique les messages `Connection refused`.

### Une question importante

Tu avais indiqué précédemment avoir modifié l'adresse du SCP en `127.0.1.10` et le service avait démarré.

Or le contenu de `/etc/open5gs/scp.yaml` que tu viens d'afficher montre de nouveau :

```yaml
address: 127.0.0.200
```

Peux-tu vérifier ce que contient actuellement le fichier avec :

```

et aussi :

```
