Tu parles probablement de `taskset` sous Linux et de la numérotation des CPU (cœurs). Il y a une confusion fréquente : **les numéros pairs/impairs ne correspondent pas forcément aux cœurs physiques/virtuels**.

### 1) Faire un `taskset`

`taskset` sert à **affiner un processus à un ou plusieurs CPU logiques**.

Voir les CPU disponibles :

```bash
lscpu
```

ou :

```bash
nproc
```

Voir l'affinité d'un processus :

```bash
taskset -p PID
```

Lancer un programme sur certains CPU :

```bash
taskset -c 0,2,4,6 mon_programme
```

Exemple : lancer un processus uniquement sur les CPU logiques 0 et 1 :

```bash
taskset -c 0,1 ./mon_programme
```

Changer l'affinité d'un processus déjà lancé :

```bash
taskset -pc 0,1 PID
```

---

### 2) Cœurs pairs = physiques, impairs = virtuels ?

**Non, ce n'est pas une règle.**

Avec l'Hyper-Threading (Intel) ou le SMT (AMD), un cœur physique possède plusieurs **threads logiques**. Linux les numérote comme des CPU logiques, mais la numérotation dépend du BIOS, du noyau et de l'architecture.

Exemple possible :

```
CPU 0  → cœur physique 0, thread 0
CPU 1  → cœur physique 0, thread 1
CPU 2  → cœur physique 1, thread 0
CPU 3  → cœur physique 1, thread 1
CPU 4  → cœur physique 2, thread 0
CPU 5  → cœur physique 2, thread 1
```

Dans ce cas, pairs et impairs sont effectivement associés au même cœur physique.

Mais une autre machine peut avoir :

```
CPU 0 → cœur 0 thread 0
CPU 1 → cœur 1 thread 0
CPU 2 → cœur 2 thread 0
CPU 3 → cœur 0 thread 1
CPU 4 → cœur 1 thread 1
CPU 5 → cœur 2 thread 1
```

Donc la règle ne tient plus.

---

### 3) Vérifier la correspondance exacte

Utilise :

```bash
lscpu -e
```

Tu verras quelque chose comme :

```
CPU NODE SOCKET CORE
0   0    0      0
1   0    0      0
2   0    0      1
3   0    0      1
4   0    0      2
5   0    0      2
```

Ici, CPU 0 et CPU 1 partagent le même cœur physique.

Une autre commande pratique :

```bash
cat /sys/devices/system/cpu/cpu*/topology/thread_siblings_list
```

Elle affiche les couples de threads qui appartiennent au même cœur physique.

---

### 4) Exemple d'utilisation pour éviter les threads virtuels

Si tu veux utiliser uniquement un thread par cœur physique, il faut d'abord identifier les CPU correspondants, puis faire par exemple :

```bash
taskset -c 0,2,4,6 ./mon_programme
```

mais **les numéros dépendent de ta machine**.

Si tu me donnes la sortie de :

```bash
lscpu -e
```

je peux te dire exactement quels CPU sont physiques et lesquels sont des threads SMT.

####################################################################
comment optimiser le scheduler de taskset car j'ai testé autres programme seul  cpu4 qui est utilisé à100% puis cpu7 à100% les autres entre cpu4 à 7 reste à 0%???

*******************************************************************

Ce que tu observes est normal avec `taskset` : **`taskset` ne fait pas de répartition de charge entre les CPU**. Il définit seulement un **ensemble de CPU autorisés**.

Exemple :

```bash
taskset -c 4-7 ./mon_programme
```

signifie :

> Le programme peut être exécuté sur CPU4, CPU5, CPU6 ou CPU7.

Cela ne signifie pas :

> Répartis automatiquement le travail à 25 % sur chacun des CPU.

Le scheduler Linux choisit un CPU parmi 4-7 selon l'état du système. Si ton programme est **mono-thread**, il peut tourner longtemps sur CPU4 à 100 %, puis migrer sur CPU7 à 100 %. Les CPU5 et CPU6 restent inutilisés car il n'existe qu'une seule tâche à exécuter.

---

### 1) Vérifier si ton programme est multi-thread

Lance :

```bash
ps -eLf | grep nom_du_programme
```

ou :

```bash
top -H
```

Si tu vois un seul thread, un seul CPU sera utilisé.

Pour utiliser 4 CPU, il faut que le programme crée plusieurs threads/processus.

---

### 2) Tester que `taskset` fonctionne bien

Teste avec un programme multi-thread :

```bash
taskset -c 4-7 stress-ng --cpu 4
```

Tu devrais voir CPU4, CPU5, CPU6 et CPU7 chargés.

---

### 3) Vérifier la topologie CPU

Il est possible aussi que tes CPU 4-7 soient des threads virtuels SMT.

Regarde :

```bash
lscpu -e
```

Exemple :

```
CPU  CORE
4    2
5    2
6    3
7    3
```

Ici CPU4 et CPU5 partagent le même cœur physique, idem pour CPU6/7.

Pour une charge performante, il vaut parfois mieux sélectionner uniquement les cœurs physiques.

---

### 4) Pour forcer un équilibrage entre threads

Si ton application utilise plusieurs threads, tu peux utiliser :

```bash
taskset -c 4-7 ./programme
```

mais aussi régler la politique CPU :

```bash
chrt -r 50 taskset -c 4-7 ./programme
```

(`chrt` change la priorité temps réel, à utiliser avec précaution.)

Pour une application SDR/UHD temps réel, on utilise souvent plutôt :

```bash
taskset -c 4-7 chrt -f 50 ./programme
```

### 5) Dans ton cas UHD

Pour UHD (USRP), le plus important n'est pas d'avoir **un CPU à 100 %**, mais d'avoir :

* un ou plusieurs cœurs dédiés,

############################################################################################
mais pourquoi avec system Monitor de linux ca dit le contraire?
**************************************************************

Bonne remarque. Si **System Monitor affiche CPU6 à 100 %** alors que `mpstat` affiche **CPU5 à 100 %**, il y a plusieurs explications possibles.

### 1) System Monitor et `mpstat` ne mesurent pas exactement de la même façon

`mpstat` lit directement les compteurs CPU du noyau Linux (`/proc/stat`) et affiche une mesure par CPU logique. C'est généralement la référence pour ce diagnostic.

System Monitor (GNOME System Monitor ou équivalent) :

* rafraîchit à intervalle régulier ;
* peut afficher une valeur légèrement décalée dans le temps ;
* peut parfois numéroter les CPU différemment selon l'interface graphique ou la configuration.

Donc un décalage CPU5 ↔ CPU6 peut arriver dans l'affichage.

---

### 2) Le thread peut migrer entre deux mesures

Le scheduler Linux peut déplacer un thread :

```
seconde 1 : CPU5 = 100 %
seconde 2 : CPU6 = 100 %
seconde 3 : CPU5 = 100 %
```

`mpstat -P ALL 1` capture une fenêtre précise de 1 seconde. System Monitor peut capturer une autre fenêtre.

Pour vérifier en temps réel :

```bash
watch -n 0.5 'ps -eLo pid,tid,psr,pcpu,comm | grep stress-ng'
```

La colonne `PSR` montrera le CPU actuel de chaque thread.

---

### 3) Vérifie aussi la numérotation dans System Monitor

Dans Linux, les CPU sont numérotés par le noyau :

```
CPU0 CPU1 CPU2 ... CPU11
```

Mais certaines interfaces affichent les cœurs physiques ou réordonnent l'affichage. Avec ton `lscpu`, tu as :

```
CPU4 + CPU5 = même cœur physique
CPU6 + CPU7 = même cœur physique
```

Donc un affichage "cœur 3" dans une interface graphique ne correspond pas forcément à "CPU3".

##########################################################*
signifie :

Les allocations mémoire de ces processus doivent venir du nœud NUMA 0.  c'est quoi NUMA 0?

***************************************************************

**NUMA 0** désigne le **nœud mémoire numéro 0** dans l'architecture NUMA de la machine.

NUMA signifie **Non-Uniform Memory Access** (accès mémoire non uniforme). C'est une architecture où les CPU sont regroupés avec des zones de mémoire qui leur sont physiquement proches.

Sur une grosse machine avec plusieurs sockets CPU, on peut avoir par exemple :

```
        Socket 0                 Socket 1
     CPU 0-15                  CPU 16-31
        |                         |
     RAM locale 0              RAM locale 1
     NUMA node 0               NUMA node 1
```

Un CPU du **nœud NUMA 0** accède plus rapidement à sa mémoire locale qu'à la mémoire du nœud NUMA 1.

---

Dans **ta machine**, d'après ton `lscpu -e` :

```
CPU NODE
0    0
1    0
2    0
...
11   0
```

Tous les CPU appartiennent à :

```
NUMA node 0
```

Donc ta machine n'a qu'un seul domaine mémoire :

```
              NUMA node 0
        +----------------------+
        | RAM                  |
        |                      |
        | CPU0 CPU1 ... CPU11  |
        +----------------------+
```

Il n'y a pas de différence de distance mémoire entre CPU4 et CPU10.

Donc cette commande :

```bash
echo 0 > /sys/fs/cgroup/gnb/cpuset.mems
```

veut simplement dire :

> "Le groupe gNB utilise la mémoire disponible du seul nœud mémoire existant."

Elle est nécessaire pour que le cpuset soit valide, mais elle n'apporte pas de gain de performance sur ta machine.

Pour ton cas (gNB 5G), les réglages importants sont plutôt :

* `cpuset.cpus = 4-11` → réserver les CPU pour gNB
* affinité des threads gNB
* éviter que l'interface graphique utilise ces CPU
* éventuellement régler les IRQ réseau/USRP sur les CPU dédiés

Le NUMA devient surtout important sur des serveurs à **2 sockets ou plus**.


##########################################################################

NUMA 0 veut dire barette numero 1?

*********************************************

Non. **NUMA 0 ne veut pas dire "barrette mémoire numéro 1".**

Un **nœud NUMA** est une **zone logique d'architecture mémoire/CPU**, pas une barrette RAM individuelle.

Dans ton cas, avec ta sortie :

```text
CPU NODE
0    0
1    0
2    0
...
11   0
```

cela veut dire :

* Tous tes CPU appartiennent au **nœud NUMA 0**
* Toute ta RAM est attachée à ce même nœud NUMA 0
* Le système ne distingue pas tes barrettes RAM une par une

Schéma simplifié de ta machine :

```
             NUMA node 0
        +--------------------+
        |                    |
        | CPU0 ... CPU11     |
        |                    |
        | RAM totale         |
        | DIMM1 DIMM2 ...    |
        +--------------------+
```

Les barrettes DIMM sont des modules physiques branchés sur la carte mère. Le BIOS et le contrôleur mémoire les regroupent derrière un ou plusieurs nœuds NUMA selon l'architecture.

Sur une machine bi-socket, on pourrait avoir :

```
NUMA node 0              NUMA node 1

CPU0-15                  CPU16-31
RAM proche               RAM proche
(DIMM A,B)               (DIMM C,D)
```

Là, on pourrait parler de mémoire locale au NUMA 0 ou au NUMA 1.

Pour voir comment ta RAM est organisée, tu peux regarder :

```bash

```

et éventuellement :

```bash

```

Mais avec ton processeur actuel, il est très probable que toutes tes barrettes soient dans **NUMA node 0**.

#################################################################################################

est ce seulement les user-space qui est cpu set 0-3?


*************************************************************

Non. Un **cpuset de cgroup ne s'applique pas uniquement au user-space**.

Un cpuset s'applique aux **processus (tâches Linux)** placés dans ce cgroup, qu'ils soient :

* des programmes user-space (`gnome-shell`, navigateur, gNB, applications...)
* des services système (`systemd`, sshd, NetworkManager, etc.)
* des démons en arrière-plan

```
 ├── thread 1
 ├── thread 2
 └── thread 3
```

tous ses threads seront limités à :

```
CPU0 CPU1 CPU2 CPU3
```

```
rcu_preempt
migration
irq/*
```

ne sont pas simplement déplacés par un cpuset utilisateur.

Pour un système temps réel (comme UHD/gNB), il faut aussi gérer :

* les IRQ matérielles
* les threads kernel
* les timers
* les services systemd

par d'autres mécanismes.

Il faut aussi nuancer : `isolcpus` est utile dans certains cas temps réel, par exemple :

* un thread critique unique affecté à un CPU précis ;
* très faible latence déterministe ;

* des CPU dédiés au gNB ;
* mais un scheduler actif à l'intérieur de ce groupe.

Oui, c'est exactement l'architecture que tu recherches. Elle utilise le **scheduler Linux normal** tout en séparant les ressources CPU par groupes.

Le principe :

```
                 Scheduler Linux normal
                         |
        +----------------+----------------+
        |                                 |
   CPU0-3                             CPU4-11
   cgroup système                     cgroup gNB
        |                                 |
  GNOME, systemd,                  gNB, DSP,
  services OS                      threads radio
```

La différence avec `isolcpus` :

* **Pas de `isolcpus`** :

  * le scheduler continue son équilibrage ;
  * un programme avec plusieurs threads sur CPU4-11 peut être réparti sur CPU4,5,6...11 ;
  * les CPU restent disponibles normalement mais avec des limites par cgroup.

* **Avec `isolcpus`** :

  * les CPU sont retirés du mécanisme d'équilibrage général ;
  * il faut souvent gérer soi-même les affinités des threads.
